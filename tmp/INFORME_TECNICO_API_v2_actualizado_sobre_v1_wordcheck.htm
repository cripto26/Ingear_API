Se cubren arquitectura, configuración, contrato de datos (schemas), modelo de persistencia (SQLAlchemy), endpoints REST (CRUD y relaciones M:N), manejo de errores, consideraciones de integración con el front y recomendaciones para endurecimiento (producción). Esta versión continúa el mismo informe y actualiza explícitamente los cambios funcionales y técnicos incorporados en la API v.2.

8. Validación, errores y códigos de respuesta

8.1. Códigos de respuesta típicos

●       200 OK: lectura/actualización exitosa.

●       201 Created: creación exitosa (POST).

●       204 No Content: eliminación exitosa (DELETE).

●       404 Not Found: recurso no existe (mensajes: 'Cliente no encontrado', etc.).

●       409 Conflict: relación duplicada (proyecto-empleado / proyecto-cliente / proyecto-despacho) o colisión de unicidad en DB.

●       422 Unprocessable Entity: validación Pydantic fallida (faltan campos requeridos o tipos incorrectos).

Códigos agregados en v2

401 Unauthorized: access token ausente, inválido o expirado.

403 Forbidden: usuario autenticado sin rol, área, cargo o permiso de vista suficiente.

502 Bad Gateway: falla de integración externa, por ejemplo Gmail, World Office, BanRep o BCE.

8.2. Validación con Pydantic

Cada recurso tiene tres esquemas Pydantic:
- *Create*: campos requeridos para crear.
- *Update*: campos opcionales (exclude_unset=True) para actualizar parcialmente.
- *Out*: modelo de respuesta con id (y campos de auditoría si aplica).

En v2 también se documentan schemas específicos para autenticación, refresh de sesión, notificaciones, tasas FX, cuentas de cobro y versiones de cotización.

8.3. Paginación

Los endpoints de listado soportan parámetros query:
- skip (int): offset (por defecto 0)
- limit (int): máximo de registros (por defecto 50)
Ejemplo: GET /api/v1/productos?skip=100&limit=50

9. Buenas prácticas y recomendaciones

9.1. Migraciones con Alembic

Actualmente el proyecto crea tablas automáticamente al iniciar. Esto es útil en desarrollo, pero en producción es mejor controlar cambios del esquema con migraciones. Recomendación:
1) Integrar Alembic.
2) Desactivar create_all en producción.
3) Ejecutar migraciones en el pipeline de despliegue.

9.2. Seguridad

●       La versión 2 ya incorpora autenticación JWT, refresh token en cookie HttpOnly, logout, cambio de contraseña y control de sesiones.

●       Mantener CORS controlado por entorno, especialmente al pasar de localhost a dominio de producción.

●       Registrar auditoría (quién cambia qué) para operaciones críticas.

9.3. Calidad de código y pruebas

●       Agregar pruebas con pytest (unitarias para CRUD y de integración para endpoints).

●       Separar configuración por entornos (dev/staging/prod).

●       Definir linters (ruff/flake8) y formateo (black) para consistencia.

9.4. Operación y seguridad v2

Proteger SECRET_KEY, Google service account, Turnstile y World Office fuera del repositorio.

Usar HTTPS, cookie Secure y SameSite adecuado en producción.

Registrar auditoría de aprobación, envío de cotizaciones, cuentas de cobro y cambios de estado.

Monitorear errores 401, 403, 409, 422 y 502 por módulo.

11. Solución de problemas (troubleshooting)

11.1. ERR_CONNECTION_REFUSED en /docs

Causa típica: el proceso de Uvicorn se cae durante startup. Revisa la consola donde ejecutaste uvicorn y corrige el error antes de volver a intentar.

11.2. Errores de conexión a PostgreSQL

Síntomas comunes: ConnectionTimeout, getaddrinfo failed, o Test-NetConnection al puerto 5432 falla.

●       Verifica que PostgreSQL esté instalado y el servicio esté iniciado.

●       Confirma usuario/clave/DB en DATABASE_URL.

●       Asegura que el puerto 5432 esté escuchando (y no bloqueado por firewall).

11.3. NoReferencedTableError (ForeignKey)

Si ves un error como 'could not find table ... with which to generate a foreign key', hay una FK apuntando a una tabla que no existe o está mal escrita. En esta base, el caso más común es Cliente.world_office_id -> world_oficce.id.

11.4. Conflictos 409 en relaciones

Al asignar un empleado/cliente/despacho a un proyecto, si la relación ya existe se devuelve 409. Solución: no repetir asignación, o eliminar primero la relación y volver a crearla.

11.5. Errores de autenticación y permisos

401: renovar sesión con /api/v1/auth/refresh o iniciar sesión nuevamente.

403: revisar área, cargo, rol inferido, permisos_vistas y marcador comercial.views.v2.

Logout/cambio de contraseña: las sesiones refresh previas pueden quedar revocadas.

11.6. Integraciones externas

World Office: validar driver ODBC, servidor, base, usuario, bodega y tiempos de consulta.

Gmail: validar service account, delegación de dominio, correo del empleado y permisos gmail.send.

FX: revisar disponibilidad de BanRep/BCE y caché de tasas.

12.3. Seguridad y sesiones

El acceso protegido se realiza con token Bearer. La sesión persistente utiliza refresh token en cookie HttpOnly y una tabla auth_refresh_session con hash del token, fechas de expiración, último uso y revocación.

El login puede validar Cloudflare Turnstile y aplica protección contra fuerza bruta mediante límites por usuario/IP, ventana de fallos y bloqueo temporal.

12.4. Cambios en persistencia

Se agregan tablas para auth_refresh_session, cuenta_cobro y notificacion.

Se conserva el modelo principal cliente, contacto, oportunidad, cotizacion, proyecto, producto, despacho y empleado.

Se mantiene el uso de tablas puente proyecto_cliente, proyecto_empleado y proyecto_despacho.

Se agrega historial de cotización en cotizacion_versiones_v2.

Se actualizan columnas de empleado para permisos_vistas y jefe_id cuando el esquema lo requiere.

12.6. Recomendaciones actualizadas

Usar HTTPS y cookies Secure/SameSite en producción.

Proteger SECRET_KEY, credenciales Google, Turnstile y World Office fuera del repositorio.

Agregar migraciones formales con Alembic para cambios de esquema.

Registrar auditoría de acciones críticas: aprobación, envío de cotización, cambio de estado y cuentas de cobro.

Agregar pruebas de endpoints para autenticación, permisos, cotizaciones, cuentas de cobro e integraciones simuladas.
